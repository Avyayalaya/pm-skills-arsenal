SCR Structure (Minto Pyramid)

Barbara Minto's Pyramid Principle structures executive communication as answer first, then grouped supporting arguments, then evidence. The SCR framework (Situation-Complication-Resolution) is the backbone. Here is how it was applied.

The SCR Breakdown

ElementPurposeTarget LengthWhat Was WrittenQuality Check
Situation Establish shared reality. No new information — just common ground. 2-3 sentences "DataVault's engine is our highest-rated feature... $45M ARR with 340 enterprise accounts... currently runs as embedded feature." PASS Shared context, no new claims
Complication Create tension. What changed that demands action. 2-3 sentences "Three competitors launched API platforms... lost 8 deals worth $2.1M... next generation of data teams will choose competitors." PASS Specific trigger, quantified stakes
Resolution State the answer. Your recommendation, upfront. 1-2 sentences Title IS the resolution: "We Should Open Our Data Transformation Engine as a Developer API Platform" PASS Recommendation in first line

The Pyramid Below the SCR

Common SCR Failures Avoided

FailureWhat It Looks LikeHow This Document Avoids It

Vague complication "The market is changing" (everything is always changing) Complication names 3 specific competitors, quantifies $2.1M in lost deals, and identifies the specific buyer segment at risk.
Resolution buried Recommendation appears on page 4 after analysis build-up The document TITLE is the resolution. A VP who reads nothing but the header knows the recommendation.
Academic structure Context → Analysis → Frameworks → Finally, conclusion Inverted: Conclusion (title) → Context (S+C) → Options → Evidence → Ask.

Ask Type Classification

Ask Type: Decision

"Approve [X]." Not alignment, not input — a specific approval of a specific investment with specific resources.

Why Not "Alignment"?

An alignment ask would be: "Do you agree we should pursue API-first strategy?" That is too vague. The PM has a specific recommendation with a specific cost — ask for the decision.

Why Not "Input"?

An input ask would be: "Should we build or partner?" That signals the PM has not done the work to recommend. The options table shows the PM has evaluated both — and recommends building.

Decision Architecture — Type 1 vs. Type 2

DimensionAssessment
Decision Type Type 1 (One-Way Door) — A public API becomes a contract with external developers. Once published and adopted, breaking changes destroy trust. Shutting down the API after customers build on it creates legal and reputational risk.
Evidence Bar High — Type 1 decisions require strong data. Met: T1 evidence on opportunity + threat. 3 design partner verbal commits.
What's Reversible Pricing model, rate limits, tier structure, SDK language priorities. These can change post-launch.
What's Not Reversible The API surface itself. Once endpoints are public and customers integrate, deprecation is a 12-18 month process with migration support.

Option Architecture

The skill requires 2-4 structurally different options. "Structurally different" means each differs on at least one fundamental dimension — not just scope or timeline.

OptionStructural AxisKey Tradeoff
A: Build API Platform Full ownership, high investment, long-term moat $2M cost + 6 months eng time. Gives up analytics polish sprint. Gains platform economics.
B: Partner with Existing API Layer Speed-to-market, low control, dependency risk $400K + live in 2 months. Gives up control over pricing, brand, and roadmap. Gains speed.
C: Do Nothing No cost, no risk, shrinking market position $0 cost. Gives up $8M+ addressable pipeline and developer segment permanently. Gains nothing — competitors embed further.
◈

Skill rule enforced: Options A and B differ structurally (build vs. partner = ownership vs. speed). Option C is "do nothing" — always included because executives need to understand the cost of inaction. The skill prevents presenting Option B as "build half the API" — that would be the same option at a different dosage, not a structural alternative.

Adversarial Self-Critique

The skill mandates identifying at least 3 genuine weaknesses in the recommendation — not pro-forma hedging, but real scenarios where the recommendation is wrong. This demonstrates intellectual honesty, which executives respect more than false certainty.

Weakness #1: The "Lost Deals" Data May Overstate API Demand

What was assumed: 8 lost deals citing "no API" in Q4 represents a structural market shift toward developer-first buying T1.

What could be wrong: Deal-loss reasons in CRM are retrospective rationalizations. Account executives may attribute losses to "no API" because it is a concrete, defensible reason — when the real reasons (pricing, feature gaps, relationship weakness) are harder to document. If 5 of the 8 losses were actually price-driven, the API investment solves the wrong problem.

Evidence that would disprove this: Independent win/loss interviews (not AE self-reports) showing that API access was a secondary factor, not primary. If DataVault conducts structured exit interviews with the 8 prospects and fewer than 5 cite API as the decisive factor, the demand signal is weaker than claimed.

Watch indicator: Post-launch API adoption by net-new customers. If new enterprise deals close but customers don't use the API within 90 days, the API wasn't the deciding factor — it was table-stakes checkbox, not actual demand.

Weakness #2: Platform Moat Thesis Rests on Analogy, Not Data

What was assumed: Opening the engine as an API creates Stripe/Twilio-like platform switching costs that compound over time T3.

What could be wrong: Stripe and Twilio are horizontal infrastructure platforms. DataVault is a vertical data transformation tool. The analogy may not hold — data transformation APIs may be more commoditized than payment or communication APIs because the output (transformed data) is fungible. A customer who integrates with DataVault's API can switch to Fivetran's API if the output schema is compatible.

Evidence that would disprove this: If API customers show the same or lower net dollar retention as dashboard-only customers after 12 months, the platform moat thesis is wrong. The switching cost mechanism (integrations create lock-in) does not operate as predicted.

Watch indicator: Net dollar retention for API customers vs. dashboard-only customers. If API NDR is not at least 10 percentage points higher after 12 months, the platform economics are not materializing.

Weakness #3: Opportunity Cost of Engineering Reallocation Is Under-Explored

What was assumed: Reallocating 4 engineers from the "analytics polish sprint" is an acceptable trade-off T3.

What could be wrong: The analytics polish sprint may be more valuable than characterized. If the sprint addresses specific customer complaints driving churn in the existing base, deprioritizing it to build an API could increase churn in the current segment while pursuing a new segment. The document mentions "analytics polish" dismissively — but that sprint may contain work that retains $4M+ in existing ARR.

Evidence that would disprove this: If the analytics polish sprint items map directly to churn-risk accounts worth more than the $3.2M at-risk API pipeline, the reallocation destroys more value than it creates. Product management should compare: pipeline at risk from no-API ($3.2M) vs. renewals at risk from delayed polish (unknown — this was not quantified in the document).

Watch indicator: Churn in existing enterprise accounts during the 6-month API build period. If churn rate increases >1.5 percentage points, investigate whether delayed analytics polish is a contributing factor.

Weakness #4: "Do Nothing" Option Is Presented as Obviously Wrong

What was assumed: Inaction leads to permanent loss of the developer segment T3.

What could be wrong: "Permanent" is a strong word. If the AI-native transformation market evolves rapidly, today's API platforms may be obsolete within 2-3 years. Building an API now could be investing in a paradigm that is about to shift. A "wait and see" strategy — while costly in the short term — preserves optionality to leapfrog directly to an AI-native API when the technology matures.

Evidence that would disprove this: If AI-native data transformation tools (e.g., LLM-powered ETL) reach production quality within 18 months, the traditional API surface we build today becomes technical debt. The $2M investment would need to be written off and rebuilt.

Watch indicator: AI-native data transformation tool launches and adoption rates. If a credible player ships LLM-powered transformation APIs that handle 80%+ of use cases, reassess whether our traditional API investment is the right bet.

◈

Self-critique quality check: Each weakness is genuine — a reasonable executive could use any of these to challenge the recommendation. The weaknesses are not pro-forma hedging ("there's always some risk") but specific scenarios with specific evidence that would invalidate specific assumptions. This is what the skill enforces.

Try It Yourself

Apply the executive-writing skill to your own documents.

Quick-Start: Write a Strategy One-Pager (90 minutes)

  1. Run the Context Gate (5 min) — Verify: right artifact? Have a recommendation? Right reader? Right timing? If any check fails, stop and address it before writing.
  2. Select format via Step 0 (5 min) — Use the routing table. Match the exec's consumption mode and the decision's complexity to one of three templates (one-pager, board memo, decision brief).
  3. Calibrate for your audience (10 min) — Look up the reader's role in the calibration matrix. Note their primary lens, evidence type, risk framing, and ask framing. Write the opening line in their frame.
  4. Draft the SCR (20 min) — Situation (2-3 sentences of shared reality), Complication (what changed), Resolution (your recommendation — this becomes the document title). If you cannot state the Resolution in two sentences, the analysis is not ready.
  5. Build options (20 min) — 2-4 structurally different options including "do nothing." For each: cost, timeline, risk (specific scenario), reversibility, key tradeoff. State your recommendation with H/M/L confidence.
  6. Compress and de-jargon (15 min) — Run the grandmother test. Translate framework names. Enforce the acronym cap (max 5). Push L2+ evidence to appendix.
  7. Write the Ask (5 min) — What you need, by when, cost of delay, next step if approved. Apply the Meeting Test: if an exec reads this and does not know what to do next, the Ask failed.
  8. Run the Quality Gate (10 min) — 12-point checklist. Fix any failures before sending. Target: 12/12. Ship at 10/12 minimum.

Common Mistakes to Avoid

Failure ModeWhat It Looks LikeHow to Fix
FM-1: Information Dump All the data is there. Charts, numbers, analysis. But no recommendation, no Ask. Exec thinks: "So what do you want me to do?" Start with the Ask. Write it first. Then build the document backward.
FM-4: Ask Buried on Page 4 Academic structure — context, analysis, frameworks, and finally the conclusion. Invert: conclusion first (Minto Pyramid). The title is the recommendation.
FM-5: Jargon Barrier "Leveraging our TAM/SAM analysis and the JTBD framework, we identified counter-positioning opportunities in the adjacent market." Grandmother test every sentence. Replace framework names with what they reveal.
FM-6: Single-Audience Doc Same document sent to CEO and CTO. CFO gets strategic framing. CTO gets financial framing. Calibrate per role. Lead with the decision-maker's frame. Others get role-specific subsections.
FM-8: Vague Risks "Risk: adoption may be slow. Mitigation: we'll monitor and adjust." State the specific scenario, its probability, its impact, and the concrete mitigation action.
